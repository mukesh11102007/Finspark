# SecureBank Anti-Fraud System: Architecture & Design

SecureBank is a prototype web application demonstrating a modern, multi-layered digital banking protection system. It integrates security telemetry, real-time client-side behavioral biometrics, and a machine learning classifier to safeguard accounts from takeover and automated transaction scripts (bots).

---

## 🏗️ Architectural Overview

The system consists of three core layers that protect every transaction request:

```mermaid
graph TD
    A[Client UI / Browser] -->|1. Mouse/Typing Telemetry| B[Telemetry Route]
    A -->|2. Transfer Request| C[Banking API Gateway]
    B -->|Logs anomaly flag| D[(SQLite DB)]
    C -->|Queries recent anomaly flags & geo-distance| D
    C -->|3. Feeds features| E[Random Forest Classifier]
    E -->|4. Predicts fraud probability > 60%| F{Blocked?}
    F -->|Yes| G[Explainable AI Context Engine]
    F -->|No| H[Complete Balance Transfer]
    G -->|Tries Ollama fallback to Rules| I[User Error Response]
```

## 🧠 Key Technologies & Mechanisms

### 1. Behavioral Biometrics Tracking
Unlike standard password/OTP layers, behavioral biometrics observe *how* an action is performed. The system flags automation (bots or scripts) by identifying:
*   **Robotic Mouse Speed & Direct Trajectories**: Humans move in curves with varying acceleration. Scripts teleport the cursor (high average speed) or move in straight lines with minimal travel distance (50-200px).
*   **Instantaneous Typing Speed**: Bots/scripts paste payloads instantly, registering typing speeds exceeding **200 WPM** (sometimes up to 1500 WPM).

### 2. Machine Learning Fraud Classifier
When a user attempts a transfer, the server checks the database for recent telemetry anomalies and user attributes:
*   **Geo-Distance calculation**: Uses the **Haversine formula** to calculate the distance between the transaction's current IP location and the user's home coordinate.
*   **Feature Alignment**: Collates parameters like `transaction_amount`, `login_attempts`, `device_risk_score`, and client biometrics (`typing_wpm`, `mouse_distance_total`) into a dataframe.
*   **Random Forest Scoring**: A pre-loaded scikit-learn classifier (`fraud_model.joblib`) computes a fraud probability score. If the score exceeds **60%**, the transaction is blocked.

### 3. Explainable AI (LLM Explanations)
To avoid generic, frustrating error messages, the system provides real-time contextual explanations:
1.  **Local LLM Check**: If Ollama is running, it queries models like `gemma3:4b` to write a single-sentence, friendly, and professional explanation based on the threat flags.
2.  **Deterministic Fallback**: If Ollama is offline, a keyword classifier matches the trigger reason (e.g., "insufficient funds", "threat", "anomaly") to return an expressive explanation template.

## 🚀 How to Run the System and Demo

### Prerequisites
Make sure you have python and standard ML packages installed:
```powershell
pip install flask flask-cors requests scikit-learn pandas numpy joblib pyopenssl
```

### Step 1: Start the Backend Server
Run the Flask server from your project workspace:
```powershell
python app.py
```
> [!NOTE]
> The server runs on **`https://127.0.0.1:5000`** with a self-signed SSL certificate (`ssl_context='adhoc'`).

### Step 2: Run the Demo Client
While the server is running, execute the provided script:
```powershell
python demo.py
```

### What the Demo Does:
1.  **Registers two dummy users** (a Sender and a Recipient).
2.  **Executes a normal transfer** under human telemetry baseline (this succeeds).
3.  **Injects anomalous telemetry** (pasted credentials simulated via `typing_speed_wpm = 650` and robotic mouse movement).
4.  **Attempts another transfer** which the ML classifier immediately flags and blocks.
5.  **Displays the AI explanation** returned by the server.
